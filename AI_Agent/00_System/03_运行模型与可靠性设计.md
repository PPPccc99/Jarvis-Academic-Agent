# 运行模型与可靠性设计

## 为什么采用 Markdown 配置

本项目的核心目标是长期辅助论文写作，而不是一次性自动生成文章。论文写作涉及大量人工判断、真实数据、仿真结果、实验约束和投稿规范，因此采用：

```text
长期角色配置 + 阶段任务单 + 知识库索引 + 输出归档 + 审稿循环
```

比采用复杂脚本或常驻进程更可靠。

## 智能体运行方式

这些智能体不是必须长期占用资源的后台程序，而是长期保存的角色说明书。每次工作时：

1. 主控智能体读取全局规则。
2. 主控智能体判断当前阶段。
3. 主控智能体读取对应子智能体配置。
4. 子智能体只读取任务相关输入。
5. 阶段输出保存为文件。
6. 后续任务优先读取阶段输出和知识库摘要。

## 模型选择原则

默认使用当前 Codex 可用的高可靠推理模型完成主控、审稿、技术判断和正文写作。若后续需要区分模型，可按任务风险分配：

| 任务 | 模型要求 |
|---|---|
| 主控调度、审稿、创新性判断 | 高推理能力、保守、能处理长上下文 |
| 文献提取、文件索引、格式整理 | 稳定、成本低、适合批处理 |
| 语言润色 | 表达自然、能保持技术含义 |
| 数据校核 | 严格、偏保守、能检查单位和逻辑 |

当前配置不硬编码具体模型名称，避免模型更新后配置失效。

## 防止重复工作

1. 目标期刊拆解结果长期保存。
2. 参考文献先分组，相关文献才深度拆解。
3. 证据库统一管理所有可引用内容。
4. 正文写作只读取必要证据条目。
5. 审稿人模拟只审当前阶段。

## 防止无限循环

以下任务最多迭代 3 轮：

- 选题与创新点审查
- 论文任务书审查
- 正文阶段审查
- 仿真和实验计划审查
- 结果分析审查

3 轮后仍未通过，主控智能体必须输出阻塞原因，而不是继续要求子智能体修改。

## 防止编造

任何输出若涉及：

- 数据
- 指标
- 对比结果
- 工程案例
- 标准
- 参考文献
- 期刊规定

必须给出来源或标注为“需用户确认”。无来源内容不得进入最终论文。

## 人工确认节点

以下节点必须由用户确认后再继续：

1. 推荐题目。
2. 最终创新点。
3. 论文章节结构。
4. 仿真和实验计划。
5. 关键结果是否可公开使用。
6. 投稿前终稿。

## 失败处理

若出现资料无法读取、CAJ 无法提取、数据不完整、结果不支撑结论、参考文献不足等问题，智能体必须输出：

```text
问题类型：
影响范围：
当前不能完成什么：
需要用户补充什么：
可替代方案：
```
