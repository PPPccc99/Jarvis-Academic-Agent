# 引言与正文反模板化写作规则

## 规则定位

本规则用于约束目标期刊拆解、选题创新点表达、任务书章节分工、正文写作、语言润色和审稿人模拟。其目的不是让论文“少写内容”，而是避免引言和正文呈现出明显的生成式文本、任务书化、答辩式自我解释或概念堆砌痕迹。

正式论文的引言应体现作者基于具体研究问题形成的自然学术推导，而不是把研究背景、已有方法、技术缺陷、评价指标、研究目标和本文工作一次性整理成过于完整、过于工整的综述段落。

## 引言抽象层级

撰写引言时，应优先采用较高层级的概括性表述。引言重在说明研究意义、核心矛盾、问题来源和研究必要性，不宜过早展开大量具体参数、特征参量、评价指标、模型条件或实验设置。

不推荐写法：

```text
其脉冲幅值、极性、到达时间、上升沿、半峰宽、频谱分布和重复特性与缺陷类型、放电过程、传播路径及测量链路状态密切相关。
```

问题在于：并列词过多，句子过长，重点分散，把适合放在理论分析或方法章节中的细节提前压入引言，削弱语言凝练性和可读性。

推荐写法：

```text
其暂态信号特征能够反映绝缘缺陷的演化状态及测量链路条件。
```

这类写法用“暂态信号特征”“演化状态”“测量链路条件”等上位概念统摄具体参数，让引言保持概括性和问题导向。后续若需要列出幅值、极性、时延、频谱等细节，应放入理论分析、方法、评价体系、仿真设置或实验章节。

## 引言自然推导逻辑

引言应按照以下逻辑逐步展开：

```text
研究背景 -> 已有研究进展 -> 现有不足 -> 本文问题 -> 本文工作
```

写作时先回答：

- 已有方法已经解决了什么？
- 在本文研究对象、场景或工况下仍有什么没有解决？
- 这个不足为什么会影响结果可靠性、工程应用或理论解释？
- 本文准备围绕哪个核心矛盾展开？
- 本文工作能够提供什么可验证的推进？

不得把很多技术点、指标和结论一次性罗列出来。引言要让读者顺着作者的问题意识自然理解研究必要性，而不是依靠堆叠概念证明研究有意义。

## 禁止模板化连续结构

除非目标期刊近期论文明确常用且语境确有必要，引言和正文应避免下列过于工整、过于完整、过于生成式的连续结构：

```text
第一、第二、第三
其一、其二、其三
一方面、另一方面、更关键的是
从三个层面来看
本文主要完成以下三个方面工作
因此，本文从以下几个方面展开
```

这类结构在任务书、开题报告或内部写作计划中可以使用，但直接出现在正式论文引言中，容易使文本显得像整理材料或由通用模板生成，而不是作者基于研究问题自然推导出的学术论述。

更合适的组织方式是：

```text
已有方法基础 -> 本文对象中的不足 -> 需要解决的核心问题
```

相关评价指标、适用条件、技术路线细节和对比方案，应转移到方法章节、评价体系章节、仿真/实验章节或结果讨论章节。

## 避免防御式和声明式边界说明

论文可以界定研究边界，也可以说明本文不重复已有工作，但不宜在引言中写成答辩式、自我解释式或免责声明式句子。

不推荐写法：

```text
本文不将……作为创新点，也不将……包装为新的物理量。
本文不讨论……，也不把……作为本文重点。
```

这类表述像开题报告自辩、答辩说明或生成式文本中的免责声明，不像正式论文引言。更合适的方式是从已有研究基础自然过渡到本文问题：

```text
已有方法已能够利用相关通道关系实现目标量表征。然而，在复杂结构及现场测量条件下，反射重叠、噪声耦合和通道响应差异可能削弱该关系的稳定性，进而影响结果重构和诊断判断的可靠性。因此，仍有必要研究复杂工况下信号处理结果的可信保持问题。
```

这种写法不是解释“本文不做什么”，而是说明已有研究做到什么、本文场景中还缺什么、本文准备解决什么。

## 正文自然表达要求

正文写作不能只追求内容完整，还应体现作者自身判断和具体研究问题下的推导过程。段落不应像通用论文模板，而应基于本文对象、证据、仿真/实验条件或技术矛盾展开。

应避免：

- 长串参数、特征、指标和名词并列。
- 过度工整的连接句式。
- “本文从三个层面”“一方面/另一方面/更关键的是”等模板化段落骨架。
- 过度解释本文为什么这样做、为什么不那样做。
- 像项目申请书、任务书、开题报告或答辩说明的表述。
- 没有具体研究对象支撑的通用判断。

应优先：

- 用上位概念概括引言中的复杂对象。
- 用具体研究场景中的核心矛盾组织段落。
- 将参数、模型、特征、评价指标后移到对应章节展开。
- 让段落围绕“问题来源”和“本文判断”自然推进。
- 保持语言凝练、克制、学术化。

## 段落自检问题

正文写作、润色和审稿时必须检查：

```text
这一段是否只是概念堆叠？
是否过早展开了应放在方法章、理论章或实验章的细节？
是否使用了模板化连接词或过于工整的编号结构？
是否像任务书、开题报告、项目申请书或答辩说明？
是否存在防御式、自我解释式、免责声明式表述？
是否体现了本文研究对象下的具体矛盾？
是否说明了已有方法解决了什么、在本文场景下还缺什么？
是否能删去部分细节后仍保持逻辑成立？
```

若段落显得太满、太细、太工整、太防御，应主动降维概括，把细节后移，并用“已有研究基础 -> 场景下不足 -> 本文问题”的方式重写。
