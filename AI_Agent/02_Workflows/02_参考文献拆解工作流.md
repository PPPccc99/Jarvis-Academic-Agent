# 参考文献拆解工作流

## 适用场景

用户希望围绕某个研究方向梳理国内外研究现状、技术路线、方法优缺点和研究空白。

启动本工作流前，主控智能体必须先确认用户参考文献文件夹的详细本地地址。不得自行在项目目录中广泛搜索参考文献。

## 与目标期刊拆解的区别

目标期刊拆解回答：

```text
这本期刊喜欢怎样组织论文？
```

参考文献拆解回答：

```text
这个研究问题已有方法做到什么程度，还有什么不足？
```

两者不得混淆。

## 步骤

### 0. 确认参考文献文件夹

必须记录：

- 参考文献文件夹详细本地地址
- 是否包含子文件夹
- 文献格式范围，例如 PDF、Word、CAJ、TXT、BibTeX
- 本轮需要拆解的主题关键词

若用户未提供文件夹，先请求用户提供，不开始扫描。

### 1. 文献分组

按本轮领域画像和主题关键词分组。下列仅为可替换示例，不得固定套用：

- 研究对象或应用场景
- 关键方法或技术路线
- 数据、样品或实验条件
- 模型、算法或理论框架
- 对比方法或基准体系
- 评价指标和验证方式
- 局限性或未覆盖场景

### 2. 单篇文献拆解

记录：

- 研究对象
- 解决问题
- 方法
- 输入数据
- 实验/仿真条件
- 评价指标
- 主要结果
- 局限性
- 可借鉴点
- 与用户课题关系

### 3. 方法对比

按方法维度建立对比表：

- 适用对象
- 所需传感器
- 数据条件
- 抗干扰能力
- 可解释性
- 工程部署难度
- 是否需要标定
- 是否适合用户当前课题

### 4. 研究空白提炼

研究空白必须满足：

- 文献中确实存在不足或未覆盖场景。
- 用户课题有可能给出方法或证据。
- 可通过本领域可接受的证据方式验证，例如实验、仿真、数据分析、理论推导、案例研究或系统综述。
- 不只是简单换对象、换参数、换软件。

### 5. 证据入库

将可用于论文写作的证据转入证据库，标注：

- 可用于引言
- 可用于方法对比
- 可用于结果讨论
- 可用于工程意义
- 仅供背景参考

## 质量要求

不得把“作者声称”直接写成“已被证明”。若文献只给出仿真结果，应标注为仿真证据；若只有实验室数据，不得扩展为现场工程验证。
