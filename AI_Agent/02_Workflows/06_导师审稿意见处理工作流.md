# 导师/审稿人意见处理工作流

## 目的

当用户已经获得导师意见、审稿意见、修改建议、批注文件或返修意见时，贾维斯必须先分析意见本身，再决定是否采纳和由哪个子智能体处理。不得让正文写作智能体直接照单全改。

## 适用输入

```text
导师文字意见
审稿人意见
Word 批注或修订稿
PDF 批注
聊天记录或邮件意见
用户整理的修改清单
期刊编辑部返修要求
```

## 基本流程

```text
用户提供意见文件或文字
-> 主控智能体建立意见处理任务
-> 文件提取与资料整理智能体提取意见文本，若为 Word/PDF 批注
-> 审稿人模拟智能体判断意见性质、合理性和优先级
-> 主控智能体生成采纳矩阵
-> 按意见类型调用对应子智能体局部修改
-> 主控智能体记录采纳/部分采纳/暂不采纳原因
```

## 意见分类

每条意见必须分类：

| 类型 | 处理智能体 |
|---|---|
| 选题或创新点质疑 | 06 选题与创新点写作智能体 + 15 审稿人模拟智能体 |
| 论文结构调整 | 07 论文任务书写作智能体 + 15 审稿人模拟智能体 |
| 引言逻辑问题 | 08 正文写作智能体 + 15 审稿人模拟智能体 |
| 方法或算法问题 | 08 正文写作智能体 + 09 仿真和实验计划安排智能体 + 15 审稿人模拟智能体 |
| 仿真或实验可行性问题 | 09 仿真和实验计划安排智能体 + 10 数据与结果校核智能体 |
| 数据、结果或结论支撑不足 | 10 数据与结果校核智能体 + 11 仿真实验结果分析智能体 |
| 图表表达问题 | 12 图表呈现智能体 |
| 语言和格式问题 | 13 语言与格式润色智能体 |
| 投稿规范问题 | 14 投稿规范检查智能体 |
| 意见本身不明确或冲突 | 01 主控智能体向用户确认 |

## 采纳等级

每条意见必须给出采纳等级：

```text
采纳：意见合理，且有证据或文本基础支持修改。
部分采纳：方向合理，但需控制表述、缩小范围或换一种实现方式。
暂不采纳：意见与数据、工程条件、目标期刊要求或论文主线冲突。
需用户确认：涉及真实数据、设备条件、作者意图、导师偏好或投稿策略。
阻塞：意见指出根本问题，当前论文无法继续推进。
```

## 采纳原则

1. 导师或审稿意见优先认真处理，但不得无证据地强行采纳。
2. 涉及真实实验、仿真结果、性能指标、设备条件的意见，必须请求用户确认。
3. 若意见要求增加不存在的数据、实验或参考文献，不得编造，只能列为“需补充材料”。
4. 若意见之间冲突，主控智能体应说明冲突并请求用户决策。
5. 若意见会改变论文核心创新点或技术路线，必须重新经过审稿人模拟智能体审查。
6. 修改应局部、精准，避免因为一条意见重写无关章节。

## 输出要求

每次处理导师/审稿意见，应输出：

```text
意见来源：
意见文件路径：
意见条目表：
采纳矩阵：
对应修改位置：
调用的子智能体：
已修改内容：
未修改内容及原因：
需要用户确认的问题：
下一步：
```

输出应保存到本轮工作文件夹：

```text
13_审稿意见
12_润色与投稿检查
08_正文草稿
```

具体保存位置由主控智能体根据意见类型决定。
